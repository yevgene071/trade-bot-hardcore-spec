# METASCALP_API_CONTRACT

Фактическая поверхность MetaScalp API, которую использует бот. Источник истины:
`../../metascalp-sdk/docs/MetaScalp-Api.md` из `metascalp-sdk` **v1.0.7**
(`8678f9d23df78c13990ccf99a20fd70f6350393f`). Всё, чего нет в этом
документе, нельзя считать доступным без отдельной live-проверки и фикстуры.

---

## 1. Базовые факты

- Host только локальный: `127.0.0.1`.
- Port range: `17845`..`17855`, один и тот же порт для HTTP и WebSocket.
- Discovery: `GET /ping` возвращает `{ "App": "MetaScalp", "Version": "..." }`.
- Все HTTP body/query поля и WS поля в документации написаны в PascalCase:
  `ConnectionId`, `Ticker`, `OrderId`, `AvgPriceFix`.
- `Ticker` в MetaScalp API — строка из каталога конкретного `ConnectionId`.
  Простые алиасы вида `BTC_USDT`/`BTCUSDT` допустимы только как compatibility
  normalization для одиночного рынка. Они не являются полной идентичностью
  инструмента.
- Для стратегий, где одновременно нужны spot и futures/perp, идентичность
  инструмента должна сохранять namespace: `ConnectionId`, exchange/account
  context, market type и точное имя тикера из MetaScalp catalog. Нельзя
  схлопывать spot и futures в один ключ `BTC_USDT`.

## 2. REST endpoints, которые можно использовать

| Метод | Path | Использование в боте |
|-------|------|----------------------|
| `GET` | `/ping` | discovery/heartbeat |
| `GET` | `/api/connections` | выбор торгового connection |
| `GET` | `/api/connections/{ConnectionId}` | health-check connection state |
| `GET` | `/api/connections/{ConnectionId}/tickers?Refresh=true\|false` | каталог тикеров и шаги цены/размера |
| `GET` | `/api/connections/{ConnectionId}/orders?Ticker=...` | только open orders конкретного тикера |
| `GET` | `/api/connections/{ConnectionId}/positions` | все open positions connection |
| `GET` | `/api/connections/{ConnectionId}/balance` | текущие balances |
| `POST` | `/api/connections/{ConnectionId}/orders` | place order |
| `POST` | `/api/connections/{ConnectionId}/orders/cancel` | cancel one order |
| `POST` | `/api/connections/{ConnectionId}/orders/cancel-all` | cancel all open orders for one ticker |
| `GET` | `/api/connections/{ConnectionId}/orderbook-snapshot?Ticker=...&ZoomIndex=...&DepthLevels=...&DepthPercent=...` | one-shot fresh book snapshot from exchange REST |
| `GET` | `/api/connections/{ConnectionId}/cluster-snapshot?Ticker=...&TimeFrame=...&ZoomIndex=...` | footprint/volume profile |
| `GET` | `/api/connections/{ConnectionId}/signal-levels?Ticker=...` | list signal levels |
| `POST` | `/api/connections/{ConnectionId}/signal-levels` | place signal level |
| `DELETE` | `/api/connections/{ConnectionId}/signal-levels/{Id}` | remove one signal level |
| `DELETE` | `/api/connections/{ConnectionId}/signal-levels?Ticker=...` | remove all signal levels for ticker |
| `DELETE` | `/api/signal-levels/triggered` | remove all triggered signal levels globally |
| `GET` | `/api/connections/{ConnectionId}/orderbook-settings?Ticker=...` | read orderbook thresholds |
| `PUT` | `/api/connections/{ConnectionId}/orderbook-settings?Ticker=...` | optional operator settings update; bot default is read-only |

Not used by core trading logic: `POST /api/change-ticker`, `POST /api/combo`.

## 3. Order API constraints

`POST /api/connections/{ConnectionId}/orders` request fields documented by API:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `Ticker` | string | yes | symbol |
| `Side` | integer | yes | `1` Buy, `2` Sell |
| `Price` | decimal | yes for non-market | Limit price for `Limit`; **trigger price** for `Stop` / `StopLoss` / `TakeProfit`; not required for `Market` |
| `Size` | decimal | yes | must be > 0 |
| `Type` | integer | no | `0` Limit, `1` Stop, `2` StopLoss, `3` TakeProfit, `4` Market |
| `ReduceOnly` | boolean | no | default `false` |

Response:

```json
{ "Status": "ok", "ClientId": "ms_limit_1234", "ExecutionTimeMs": 123.45 }
```

Important constraints:

- `POST /orders` response says `ClientId` is auto-generated by MetaScalp.
  The `GET open orders` table calls `ClientId` client-generated, but docs do
  not show any request field for caller-supplied `ClientId`; treat it as an
  opaque server value until a contract test proves otherwise. **Do not send**
  `ClientId`, `ClientOrderId`, or idempotency-key fields in the JSON body.
- `GET /api/connections/{ConnectionId}/orders?Ticker=...` requires only
  `Ticker` and returns open orders. Docs do not show
  `Status`, timestamp, side/type/price/size filters, or closed-order history.
- `GET /api/connections/{ConnectionId}/orders?Ticker=...` response may contain `ClientId`, `TriggerPrice`, `CreateDate`;
  WS `order_update` does not document `ClientId` or `TriggerPrice`.
- Robust idempotent retry after HTTP timeout is therefore not fully solvable
  with the documented API. If timeout/5xx happens after the request body may
  have been sent and reconciliation finds no order, live code must not send a
  blind second POST; it must enter `SubmitUnknown`, pause the ticker, and require
  recovery/operator ack. Retry is only safe when the HTTP layer proves the body
  was not sent.
  *   **Компенсаторный механизм для `SubmitUnknown`:** В случае перехода в состояние `SubmitUnknown`, бот должен не просто "паузить тикер", а активировать специализированный механизм разрешения неопределенности. Это может включать:
      1.  **Периодический опрос открытых ордеров:** `GET /api/connections/{ConnectionId}/orders?Ticker=...` с экспоненциальной задержкой.
      2.  **Сравнение с локальным состоянием:** Если ордер найден, его `OrderId` должен быть сопоставлен с ожидаемым локальным `OrderId` (если таковой есть) или сгенерирован новым уникальным идентификатором для отслеживания.
      3.  **Таймаут и ручное вмешательство:** Если ордер не найден в течение заданного таймаута (например, 30 секунд), инициировать алерт для оператора и перейти в режим "только чтение" для данного тикера, ожидая ручного разрешения.
      4.  **Idempotency Key (если доступен):** Если MetaScalp API в будущем добавит поддержку `clientOrderId` или `Idempotency-Key` в заголовках, этот механизм должен быть приоритетным для обеспечения идемпотентности.
- SDK v1.0.7 documents server-side stop/TP encoding: for `Stop`,
  `StopLoss`, and `TakeProfit`, `Price` is the trigger price. `Stop` is
  documented as stop-limit with MetaScalp/UI-defined offset; `StopLoss` and
  `TakeProfit` are stop-market close orders. Demo contract tests are still a
  live gate because exchange-specific behaviour and account settings can vary.

## 4. WebSocket subscriptions

SDK v1.0.7 exposes a single documented WS endpoint:
`ws://127.0.0.1:{port}/`. Do not use legacy/synthetic paths such as
`/notifications`; app notifications and signal-level streams are normal
messages on the same endpoint.

All WS messages use:

```json
{ "Type": "message_type", "Data": { "...": "..." } }
```

Supported client messages:

| Type | Data | Events received |
|------|------|-----------------|
| `subscribe` | `{ "ConnectionId": 123 }` | `order_update`, `position_update`, `balance_update`, `finres_update` |
| `trade_subscribe` | `{ "ConnectionId": 123, "Ticker": "BTCUSDT", "ZoomIndex": 1 }` | `trade_update` |
| `orderbook_subscribe` | `{ "ConnectionId": 123, "Ticker": "BTCUSDT", "ZoomIndex": 0, "DepthLevels": 50, "DepthPercent": 0.5, "FetchSnapshot": true }` | `orderbook_snapshot`, `orderbook_update` |
| `mark_price_subscribe` | `{ "ConnectionId": 123, "Ticker": "BTCUSDT" }` | `mark_price_update` |
| `funding_subscribe` | `{ "ConnectionId": 123, "Ticker": "BTCUSDT" }` | `funding_update` |
| `notification_subscribe` | `{}` | `notification_snapshot`, `notification_update` |
| `signal_level_subscribe` | `{}` | `signal_levels_snapshot`, `signal_level_placed`, `signal_level_triggered`, `signal_level_removed`, `signal_levels_removed_all`, `signal_levels_removed_triggered` |

There is no documented `finres_subscribe`: `finres_update` comes from the
connection-level `subscribe`.

`orderbook_subscribe` optional fields:

- `ZoomIndex`: aggregation factor for snapshot and updates.
- `DepthLevels`: top-N levels per side, applied to **snapshot only** after
  zoom/percent filtering.
- `DepthPercent`: per-side band anchored on best ask / best bid; applies to
  snapshot and subsequent updates. `BestAsk` / `BestBid` are never filtered.
- `FetchSnapshot=false`: skip initial exchange REST snapshot for mass
  subscriptions. Seed state separately with REST `orderbook-snapshot` when
  needed.

## 5. WS payload facts

- `order_update`: `ConnectionId`, `OrderId`, `Ticker`, `Side`, `Type`, `Price`,
  `FilledPrice`, `Size`, `FilledSize`, `Fee`, `FeeCurrency`, `Status`, `Time`.
- `position_update`: `ConnectionId`, `PositionId`, `Ticker`, `Side`, `Size`,
  `AvgPrice`, `AvgPriceFix`, `AvgPriceDyn`, `Status`.
- `balance_update`: `ConnectionId`, `Balances[].{Coin, Total, Free, Locked}`;
  docs state it is debounced about 500 ms.
- `finres_update`: `ConnectionId`,
  `Finreses[].{Currency, Result, Fee, Funds, Available, Blocked}`.
- `trade_update`: `ConnectionId`, `Ticker`, `Trades[].{Price, Size, Side, Time}`.
- `orderbook_snapshot`: `Asks`, `Bids`, `BestAsk`, `BestBid`.
- `orderbook_update`: `Updates[].{Price, Size, Type}`, where `Size=0` removes a level.
- `mark_price_update`: `ConnectionId`, `Ticker`, `MarkPrice`.
- `funding_update`: `ConnectionId`, `Ticker`, `FundingRate`, `FundingTime`.

Consequences:

- `AvgPriceFix`/`AvgPriceDyn` are position fields, not order fields.
- `balance_update` has no `OrderId`, `ClientId`, or local reservation id. Balance
  reservation reconciliation cannot match a balance event to a specific order by id.
- `position_update` has no timestamp in docs; do not resolve races by
  `position_update.ServerTime`.
- `Finreses[].Result` is documented as profit/loss since connection was
  initialized; daily PnL must use a persisted day-start baseline, not sum
  `finres_update` events.
- `trade_update` may be server-aggregated by the order book setting
  `AddingTicksForAPeriod` (default about 200 ms). Raw exchange trades require
  setting that value to `0` for the `(connection, ticker)`.

## 6. Notifications and signal levels

`notification_subscribe` is app-wide and has no `ConnectionId` filter. Notification
payload fields include `ExchangeId`, `MarketType` (string), `Ticker`, `Price`,
`Size`, `Date`. Client-side filtering can only use those documented fields.

Notification types documented:

- `Trade`
- `SignalLevel`
- `BigOrderBookAmount`
- `BigOrderBookAmount2`
- `BigTick`
- `ScreenerNewCoin`

Signal levels:

- Place request is only `{ "Ticker": "...", "Price": ... }`.
- Trigger rule is determined automatically from current best ask.
- Order book data must already be active for that ticker or place fails with
  "No market data... Subscribe to order book data for this ticker first."
- Canonical event stream for `signal_level_triggered` is `signal_level_subscribe`,
  not `notification_subscribe`.

## 7. REST order book snapshot

`GET /api/connections/{ConnectionId}/orderbook-snapshot?Ticker=...` performs
one fresh exchange REST request. It has no cache lookup and no WebSocket
subscription side effects.

Supported query parameters:

| Field | Required | Notes |
|-------|----------|-------|
| `Ticker` | yes | symbol |
| `ZoomIndex` | no | price aggregation, default `0` |
| `DepthLevels` | no | top-N levels per side, must be `>= 1` |
| `DepthPercent` | no | per-side band, must be `> 0` |

Response is the same shape as WS `orderbook_snapshot` plus `UpdateId`.
`501 Not Implemented` means the exchange does not expose a REST snapshot
endpoint; the bot must fall back to WS snapshot subscription for that ticker.

---

## 8. REST rate limits

MetaScalp docs do not specify REST rate limits for `cluster-snapshot`, `tickers`,
`orderbook-snapshot`, `orders`, or `orderbook-settings`. The bot must use
conservative client-side limits (`poll_jitter_sec`, `max_concurrent_requests`)
and measure live latency/error rates before increasing polling.

## 9. Required contract tests before live

- **T4-STOP-CONTRACT-TEST (блокер):** Place `Type=Stop`, `Type=StopLoss`, and
  `Type=TakeProfit` on demo using `Price=X` as documented trigger price; verify
  `GET /api/connections/{ConnectionId}/orders?Ticker=...` reports
  `TriggerPrice == X` where applicable and WS events arrive as expected.
  **Если тест не пройден — live заблокирован.**
- Verify `GET /orderbook-snapshot` for the selected exchange. If it returns
  `501`, mark the ticker/exchange as WS-snapshot-only and do not use
  `FetchSnapshot=false` without another seed source.
- Verify `orderbook_subscribe` with `FetchSnapshot=false` does not emit an
  initial snapshot to the first subscriber and that later snapshot requests
  resync cleanly.
- On HTTP timeout simulation, verify whether open-order reconciliation can find
  accepted limit orders. **Для market-ордеров:** дополнительно проверить
  `GET /positions` — market исполняется мгновенно и не появляется в open orders.
  Assert live code enters `SubmitUnknown` for unresolved cases instead of retrying.
- Verify `balance_update` behavior after two orders within 500 ms; reservation
  logic must remain aggregate, not per-order matched.
- Verify `finres_update` arrives after connection-level `subscribe`; no
  separate subscription should be attempted.
- Verify `mark_price_update` and `funding_update` arrive on perpetual futures
  where the exchange supports them; absence after ack is valid for unsupported
  markets.
- Verify `signal_level_subscribe` delivers `signal_level_triggered` and that
  `DELETE /api/signal-levels/triggered` emits `signal_levels_removed_triggered`.

## 10. REST positions — ограничение AvgPriceFix

`GET /api/connections/{ConnectionId}/positions` возвращает `AvgPrice`, но **не**
`AvgPriceFix` / `AvgPriceDyn` (эти поля есть только в WS `position_update`).

Последствия:
- При startup recovery (T4-RECOVERY) и WS-loss polling бот не может узнать
  точную `AvgPriceFix` для BE-стопа через REST.
- Workaround: использовать `AvgPrice` из REST как приближение (совпадает с
  `AvgPriceFix` пока нет partial exit). После первого WS `position_update`
  обновить значение на настоящий `AvgPriceFix`.
- **Если позиция уже была частично закрыта** до рестарта — `AvgPrice` ≠
  `AvgPriceFix`, BE-стоп будет неточным до восстановления WS. В этом случае
  emergency-stop на `max_recovery_stop_bps` защищает от убытка.
