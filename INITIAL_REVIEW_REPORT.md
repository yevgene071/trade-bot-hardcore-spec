# Отчёт по критическому ревью документации торгового бота (MetaScalp C++ Bot)

## 1. Резюме обзора
Документация представляет собой попытку создать **высокочастотную (HFT) инфраструктуру** для реализации **ручных скальперских стратегий**. Проект страдает от классической проблемы «over-engineering»: заявленные технические средства (SIMD, lock-free, Bayesian estimators) избыточны для тех скоростей, которые может обеспечить связка «MetaScalp + Binance/Bybit» через локальный REST/WS шлюз. 

**Вердикт:** Проект в текущем виде — это «трактор с двигателем от болида F1», который рискует никогда не выйти из стадии разработки (Phase 0-1) из-за колоссальной инженерной нагрузки.

## 2. Критические риски и «Дыры»

### 2.1. Иллюзия безопасности MetaScalp API (Блокер)
В документе `METASCALP_API_CONTRACT.md` и `README.md` (OQ-1, OQ-2) признано, что **идемпотентность ордеров не гарантирована**. 
*   **Проблема:** Бот планирует использовать `LiveExecutor` с логикой «SubmitUnknown» при таймаутах. В условиях скальпинга на волатильности (пробои), задержка в 500-1000 мс на «выяснение судьбы ордера» превращает прибыльную сделку в убыточную.
*   **Риск:** Дублирование ордеров при сетевых сбоях или «зависание» позиции без стопа. MetaScalp API не предоставляет `clientOrderId`, что для C++ бота такого уровня — критический недостаток.

### 2.2. Архитектурный Over-Engineering (Phase 0/1)
План разработки (Roadmap) отводит 3 недели на «Фундамент» (T0-PERF, T0-NUMERIC). 
*   **Критика:** Использование `xsimd` для суммирования стакана и `arena allocators` для объектов тикера — это преждевременная оптимизация. Бутылочным горлышком будет не CPU, а задержка обработки JSON в MetaScalp (который сам является прослойкой) и сетевой лаг до биржи (30-100 мс). 
*   **Последствие:** Разработчик потратит 2 месяца на отладку `lock-free` очередей и `AVX-512` инструкций, так и не дойдя до логики входа в сделку.

### 2.3. Математическая избыточность сигналов
В `SIGNAL_DETECTION.md` предлагаются:
*   Bayesian evidence accumulation для айсбергов.
*   Hawkes processes и CUSUM для ленты.
*   DEMA вместо SMA для всех порогов.
*   **Проблема:** Скальпинг на крипте — это грязные данные с огромным количеством манипуляций (spoofing). Сложные модели на таких данных склонны к переобучению (overfitting). Простые эвристики из `Мануал по стратегии.docx` будут работать надежнее, чем Байесовский фильтр, который «сойдёт с ума» от одного крупного спуфера.

### 2.4. Риск-менеджмент: «Морж» и WS-Loss
*   **R14 (Hard Kill):** Бот планирует закрывать позицию по рынку через REST, если убыток > 1.5%. 
*   **Дыра:** В моменты сильных пролив (flash crash) REST API бирж часто «ложится» первым. Если WebSocket упал (как описано в R13/R14), а REST не отвечает, бот становится беспомощным зрителем слива депозита. В документации отсутствует описание **аварийного внешнего мониторинга** (независимого от основного процесса).

## 3. Анализ стратегий (STRATEGIES.md vs DOCX)

| Аспект | В мануале (DOCX) | В спецификации (MD) | Риск / Ошибка |
| :--- | :--- | :--- | :--- |
| **Вход в пробой** | «Когда лента ускоряется и плотность разъедают» | `DensityEating` (50% объема за 3с) | Фиксированные 50% — слишком грубо. На разных монетах «активность» разная. |
| **Стоп-лосс** | «Короткий, за плотность» | `max_stop_bps = 20` (0.2%) | 0.2% на альтах — это гарантированный вынос рыночным шумом. |
| **Поводырь** | «Биток не должен идти против» | Pearson Correlation (Welford) | Корреляция — запаздывающий индикатор. Трейдеры смотрят на *динамику* (импульс), а не на коэффициент за 30 секунд. |

## 4. Рекомендации (Жёсткие правки)

1.  **Упростить Phase 0-1:** Выкинуть `xsimd`, `arena allocators` и `lock-free` на старте. Использовать стандартные контейнеры и `std::mutex`. Если бот не может заработать на «медленном» C++, он не заработает и на «быстром».
2.  **Решить вопрос с API:** Если MetaScalp не поддерживает `clientOrderId`, проект должен либо сменить шлюз на прямой коннектор к Binance (CCXT C++ или кастомный), либо внедрить жесточайший лимит на количество активных «неподтвержденных» ордеров (не более 1 на весь бот).
3.  **Добавить «Черный ящик»:** Реализовать независимый watchdog-процесс на Python/Go, который имеет свои ключи API и только одну задачу: «Если основной бот замолчал на 10 секунд, а позиция открыта — закрыть всё на бирже».
4.  **Калибровка вместо Формул:** Заменить Байесовские детекторы на простые счетчики с динамическими порогами (напр. `size > 5 * average_volume`), где `average_volume` считается за последние 5 минут.
5.  **Документарный разрыв:** В архиве отсутствуют файлы `config.toml` и `news_calendar.json`, на которые ссылается `TASK_SPECS.md`. Без примеров конфига спецификация неполная.

## 5. Итоговая оценка готовности
*   **Логика:** 8/10 (глубокое понимание скальпинга).
*   **Архитектура:** 4/10 (переусложнено, не соответствует возможностям API).
*   **Безопасность:** 3/10 (критическая зависимость от стабильности локального шлюза и WebSocket).

**Рекомендация:** Остановить разработку «идеальной инфраструктуры» и сделать MVP, который просто входит в сделку и ставит стоп. 80% заявленного кода в `TASK_SPECS.md` не принесут прибыли, но увеличат время разработки на порядок.
