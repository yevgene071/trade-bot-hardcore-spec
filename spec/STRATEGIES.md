# STRATEGIES

Формализация торговых стратегий, полученных из методических материалов
(`strategies_raw.txt` + видео-мануалы). Каждая стратегия описана по единой
схеме:

1. **Идея и контекст** — что ловим, в каком рынке работает
2. **Предусловия** — какой фон и инструмент допускается
3. **Вход (entry)** — композиция сигналов, пороги, цена входа
4. **Стоп (stop)** — куда и почему (важно: короткий стоп)
5. **Цели (TP1/TP2)** — куда идём и как выходим частями
6. **Инвалидация** — что убивает план ДО входа и ПОСЛЕ входа
7. **Типичные ошибки** — чего избегаем

Используемые детекторы и их пороги — см. `SIGNAL_DETECTION.md`.
Структуры данных (`TradePlan`, `Signal`, `FeatureFrame`) — см. `ARCHITECTURE.md`.

Все параметры (размеры, таймауты, коэффициенты R) берутся из конфига —
см. `config.toml` раздел `[strategies.*]`. Числа в документе — **дефолты
для крипто-фьючерсов** (BTC/ETH/SOL USDT perpetual).

## 0. Общие принципы (для всех стратегий)

### 0.0. Кто решает, какую стратегию запускать на каком тикере

Список торгуемых тикеров **не задаётся вручную**. `TickerUniverse`
(см. `ARCHITECTURE.md § 2.11`) собирает pool кандидатов из биржи
(`GET /api/connections/{id}/tickers`),
прогоняет warmup-статистику (ликвидность/спред/волатильность) и для каждого
тикера считает **affinity score** под каждую стратегию. Стратегия активируется
на тикере, только если его affinity ≥ `affinity_threshold` соответствующей
стратегии.

Каждая стратегия ниже имеет раздел **«Профиль кандидата (affinity)»** —
объективные критерии, которые TickerUniverse использует. Конфиг-секции —
`[universe.affinity.<strategy>]`.



### 0.1. Короткий стоп — грааль

> Если в трейдинге и есть Грааль, то имя ему «Короткий Стоп».
> (из мануала)

Любой вход обязан иметь явное основание для стопа — объект в рынке, об
который мы можем выйти: плотность, айсберг, край уровня, локальный экстремум.
**Вход без объекта для стопа запрещён** — такие TradePlan отбрасываются
ещё на уровне StrategyEngine.

### 0.2. Асимметрия R:R

- Минимум TP1 должен быть **≥ 1R** от входа (где R = дистанция до стопа)
- TP2 — желательно **≥ 2R**
- Если в плане TP1 < 1R → план отбрасывается

### 0.3. Частичный выход

- На TP1 закрываем **часть позиции** — параметр `tp1_size_ratio` **per-strategy**
  в секции `[strategies.<name>]` (defaults: bounce=0.5, breakout=0.5, leaderlag=0.6, flush=0.7)
- После TP1 стоп переводится в безубыток (`move_stop_to_breakeven_after_tp1 = true`)
  — используется `AvgPriceFix` (см. ARCH § 2.8) как цена BE
- На TP2 — закрываем остаток

### 0.4. Таймаут плана

- Если вход не произошёл за `entry_timeout_sec` (default **10 сек**)
  от генерации плана — план инвалидируется, ордер отменяется
- Если после входа позиция не двигается в нашу сторону за
  `no_progress_timeout_sec` — закрываем по текущей цене. **Per-strategy default:**
  - BounceFromDensity = **120** сек (медленная стратегия, отскок может затянуться)
  - BreakoutEatThrough = **60** сек (средняя скорость, но после пробоя ожидаем движение)
  - LeaderLag = **15** сек (быстрая, лаг 2–5 сек, если не поехал за 15 = не поедет)
  - FlushReversal = **90** сек (разворот после паники, нужно время)

### 0.5. Один вход — один TradePlan

Стратегия не генерирует новый план, пока предыдущий по тому же тикеру/направлению
в состоянии `PendingEntry` или `Open`.

### 0.6. Проверки перед отправкой в Executor

Любой TradePlan проходит через `RiskManager` (см. `RISK_MANAGEMENT.md`):
- объём в пределах лимита
- число одновременных позиций не превышено
- дневной лимит убытка не превышен
- тикер в активном универсе (`TickerUniverse::is_tradable`, см. `ARCHITECTURE.md § 2.11`)
- news blackout (R12) и funding blackout (R13) пройдены

### 0.7. Affinity score — формальная модель

Affinity score тикера под стратегию вычисляется как **взвешенная сумма
мягких критериев**, ПРИ УСЛОВИИ что все **hard-критерии** выполнены:

```
score(ticker, strategy) =
    if any(hard_criteria failed): 0             # gate
    else: sum( weight_i * pass_i )               # pass_i ∈ {0,1}
```

Список критериев и их веса — в таблицах каждой стратегии (§ 1.2, § 2.2,
§ 3.2, § 4.2). **Hard** помечены явно — они работают как gate (один fail →
score = 0). **Soft** — нормируют score в [0,1]. Тикер допущен в
стратегию, если `score ≥ affinity_threshold` **непрерывно** в течение
`affinity_stable_min` (default 5 мин).

Это устраняет ошибку прежней формулы «0.25 × N» — теперь у каждого
критерия явный вес, и сумма весов soft-критериев = 1.0 по построению.

**Warmup penalty для исторических метрик:** soft-критерии, основанные на
7-дневной статистике (`*_7d`), имеют `warmup=0` — при отсутствии
истории их вес в сумме **обнуляется** и перераспределяется пропорционально
между остальными. Это снижает общий score и затрудняет прохождение
`affinity_threshold`. Опционально: `require_7d_history = true` (default **false**,
конфиг per-strategy) — hard-gate: без 7д статистики стратегия вообще не включается.

```
score(ticker, strategy) =
    if any(hard_criteria failed): 0
    available_soft = [c for c in soft_criteria if c.has_data()]
    total_weight = sum(c.weight for c in available_soft)
    if total_weight == 0: 0
    else: sum(c.weight / total_weight * c.pass for c in available_soft)
```

## 1. BounceFromDensity — Отскок от плотности

### 1.1. Идея

Цена импульсно подходит к уровню, на уровне стоит крупная плотность
(сторона противоположна нашему направлению), агрессия в ленте затухает —
значит продавцы/покупатели не могут пробить. Ждём разворот.

### 1.2. Профиль кандидата (affinity) — `[universe.affinity.bounce]`

| Критерий | Kind | Weight | Default | Зачем |
|----------|------|--------|---------|-------|
| `min_volume_24h_usd` | **hard** | — | $100M | без ликвидности нет «упругого» поведения у плотности |
| `max_avg_spread_bps` | **hard** | — | 3 | широкий спред съедает 1R, стратегия не работает |
| `require_low_news_density` | **hard** | — | true | в ближайший час нет high-importance новостей |
| `min_density_events_per_hour` | soft | **0.50** | 4 | сам факт того, что в стакане **возникают** плотности (DensityDetector) |
| `min_levels_in_60min` | soft | **0.30** | 3 | LevelDetector нашёл хотя бы 3 валидных уровня |
| `min_bounce_success_rate_7d` | soft | **0.20** | 0.40 | за последние 7 дней ≥ 40% плотностей удержались (отскок состоялся); warmup-значение 0 до накопления статистики |
| `affinity_threshold` | — | — | 0.6 | итоговый порог score |
| `affinity_stable_min` | — | — | 5 | мин непрерывного удержания score ≥ threshold |

Сумма soft-весов = 1.0. Hard-критерии действуют как gate (любой fail ⇒
score = 0, независимо от soft).

### 1.3. Предусловия (раннтайм-проверки на каждый план)

- Тикер допущен стратегией в `TickerUniverse` (`is_tradable_by(t, "bounce")` — см. `ARCHITECTURE.md § 2.11`)
- Спред `<= max_spread_bps` (default **3 bps**)
- Не позднее `max_level_age_sec` после последнего касания уровня (default 60 сек)
- Поводырь НЕ движется против нас за последние 5 сек (|`leader_change_5s`| < 0.2% в противоположную сторону)

### 1.4. Условия входа (композиция сигналов)

Одновременно должны быть выполнены **ВСЕ**:

| Условие | Сигнал/метрика | Порог |
|---------|----------------|-------|
| C1 | `LevelApproach` к уровню | `dist_to_level_bps <= 10` |
| C2 | `ApproachAnalyzer` классифицирует подход как `impulse` | `speed >= 0.8 bps/sec`, `pullbacks <= 1` |
| C3 | `DensityDetected` на уровне, противоположная нашему направлению сторона | возраст плотности ≥ **5 сек**, размер ≥ `10× avg_level_size` |
| C4 | `TapeFade` — агрессия в сторону уровня затухает | `current_rate <= 0.2 × peak_rate` |
| C5 | НЕТ `IcebergSuspected` на уровне (иначе это breakout-сетап, не bounce) | evidence_count = 0 |
| C6 | НЕТ `LeaderMove` против нашего направления за 3 сек | \|leader_change_3s\| < 0.2% contra |

**Направление сделки:**
- Плотность в **ask** → покупать (long, bounce вниз → вверх)
- Плотность в **bid** → продавать (short, bounce вверх → вниз)

### 1.5. Параметры TradePlan

- **entry_type** = `Limit`
- **entry_price** =
  - для long: `level.price + entry_offset_bps` (default **3 bps** выше уровня, чтобы встать В плотность покупок, как описано в мануале: "выставляемся ближе к плотностям")
  - для short: `level.price - entry_offset_bps`
- **stop_price** = с ОБРАТНОЙ стороны плотности:
  - для long: `density_price - stop_buffer_bps` (default **5 bps** за плотностью)
  - для short: `density_price + stop_buffer_bps`
- **tp1_price** = `entry_price ± 1.5R` (default)
- **tp2_price** = опционально `entry_price ± 3R` при наличии следующего видимого уровня
- **tp1_size_ratio** = 0.5
- **size_coin** = из `RiskManager.compute_size(risk_usd, stop_distance)`

### 1.6. Инвалидация до входа

Любое из событий ⇒ план снимается, ордер отменяется:
- `DensityRemoved` (плотность снята)
- появление `IcebergSuspected` на том же уровне
- `TapeBurst` в сторону уровня сильнее `burst_ratio = 4` (прорыв начинается)
- `LeaderMove` против нашего направления с `|lag_pct| >= 0.3%`

### 1.7. Инвалидация после входа

- сработал стоп
- плотность снята ДО достижения TP1 → закрыть позицию по рынку, не ждать стоп
- `no_progress_timeout_sec` (default **120 сек** для bounce, см. § 0.4)

### 1.8. Типичные ошибки

- **Вход на "свежую" плотность (возраст < 5 сек)** — часто ложная, манипуляция
- **Игнорирование поводыря** — если BTC валит вниз, отскок по альту в long крайне рискован
- **Слишком широкий стоп** — если за плотностью нет другого уровня ближе `stop_anchor_max_bps` (конфиг `[strategies.bounce]`, default **15 bps**), стоп получится слишком широким. Стратегия отказывается от сделки.

## 2. BreakoutEatThrough — Пробой через проедание плотности

### 2.1. Идея

Плотность активно разъедается агрессивными сделками по рынку. Уровень
вот-вот будет пробит. Входим **вместе с агрессором** — как только видим
признаки того, что плотность доедается, — с расчётом на ускорение
после пробоя (сработают стопы встречной стороны).

### 2.2. Профиль кандидата (affinity) — `[universe.affinity.breakout]`

| Критерий | Kind | Weight | Default | Зачем |
|----------|------|--------|---------|-------|
| `min_volume_24h_usd` | **hard** | — | $100M | для realistic ликвидности после пробоя |
| `max_avg_spread_bps` | **hard** | — | 4 | чуть шире, чем bounce — приоритет скорости |
| `min_volatility_1min_bps` | **hard** | — | 8 | без волатильности нет ускорения после пробоя |
| `min_density_events_per_hour` | soft | **0.30** | 3 | без плотностей нет «уровней для пробоя» |
| `min_eating_events_per_hour` | soft | **0.40** | 1 | DensityEating (см. SIGNAL § 1.1) — иначе паттерна нет |
| `min_follow_through_rate_7d` | soft | **0.20** | 0.35 | ≥35% пробоев уходят в follow-through за 10 сек; warmup=0 |
| `min_breakout_events_per_day` | soft | **0.10** | 5 | паттерн пробоя наблюдается регулярно |
| `affinity_threshold` | — | — | 0.55 |  |
| `affinity_stable_min` | — | — | 5 |  |

### 2.3. Предусловия (раннтайм-проверки на каждый план)

- Тикер допущен стратегией в `TickerUniverse` (`is_tradable_by(t, "breakout")`)
- Уровень имеет статус `LevelFormed` и последнее касание было неудачной попыткой пробоя ИЛИ классификация подхода = `slow` (накопление)

### 2.4. Условия входа

| Условие | Сигнал/метрика | Порог |
|---------|----------------|-------|
| C1 | Плотность **активно разъедается** (`DensityEating`) | за последние **3 сек** через уровень прошло ≥ **50% от изначального размера** плотности |
| C2 | `TapeBurst` в сторону пробоя | `burst_ratio >= 3.0`, `burst_min_volume_usd >= $100k` |
| C3 | Поддержка сзади: либо `DensityDetected` на нашей стороне в пределах 20 bps от входа, либо `IcebergSuspected` на нашей стороне | — |
| C4 | Поводырь **соглашается** или нейтрален | `sign(leader_change_5s) == sign(our_direction)` ИЛИ `\|leader_change_5s\| < 0.1%` |
| C5 | Не ближе `min_distance_from_best_bps` (**2 bps**) от best — иначе опоздали | — |

### 2.5. Параметры TradePlan

- **entry_type** = `Market` (скорость критична; допустим, слиппедж заложен)
  ИЛИ `Limit` с ценой `mid + aggressive_offset_bps` (default **2 bps** в сторону движения)
- **stop_price** = за плотностью поддержки (из C3):
  - для long: `support_density_price - stop_buffer_bps`
  - если поддержки нет в радиусе 25 bps от входа → **стратегия отказывается от плана**
- **tp1_price** = `entry_price ± 1.5R` (но не ближе следующего видимого уровня)
- **tp2_price** = `entry_price ± 3R` если есть целевой уровень
- **tp1_size_ratio** = 0.5

### 2.6. Инвалидация до входа

- `DensityEating` прекратился (плотность возобновила размер ≥ 80% исходного) → пробой не состоялся, bounce → отказ
- `TapeFade` — агрессия угасла
- `LeaderMove` против нашего направления

### 2.7. Инвалидация после входа

- Стоп
- Цена за `post_entry_grace_sec` = **5 сек** не прошла `min_follow_through_bps` = **10 bps** от входа → закрыть по рынку (ложный пробой, он же "пробой = неудавшийся отскок")

### 2.8. Типичные ошибки

- **Вход на айсберг-маскировке.** Если то, что мы считали плотностью, оказалось айсбергом, цена не пойдёт дальше. Поэтому в этой стратегии важно, чтобы `IcebergDetector` **НЕ** маркировал уровень как айсберг.
- **Вход без поддержки сзади.** Если нет плотности/айсберга для стопа — R неизвестен, сделка запрещена.

## 3. LeaderLag — Вход по отставанию от поводыря

### 3.1. Идея

Поводырь (BTC для альта) сдвинулся существенно, а наш инструмент ещё не
подтянулся. У нас есть **короткое окно** (1–3 сек) войти до того, как
цена нашего инструмента закроет гэп корреляции.

### 3.2. Профиль кандидата (affinity) — `[universe.affinity.leaderlag]`

| Критерий | Kind | Weight | Default | Зачем |
|----------|------|--------|---------|-------|
| `require_leader` | **hard** | — | `"BTCUSDT"` | без поводыря не работает |
| `exclude_self_for_leader` | **hard** | — | true | сам BTCUSDT не торгуется этой стратегией |
| `min_correlation_60s` | **hard** | — | 0.6 | без корреляции сигнал недостоверен |
| `min_volume_24h_usd` | **hard** | — | $50M | минимум ликвидности |
| `max_avg_spread_bps` | **hard** | — | 3 | для узкого стопа |
| `min_leader_lag_events_per_hour` | soft | **0.50** | 2 | паттерн «лаг» наблюдается регулярно |
| `avg_catchup_completion_rate_7d` | soft | **0.30** | 0.50 | ≥50% лагов закрывались за `lag_max_age_ms * 2`; warmup=0 |
| `leader_lag_consistency_7d` | soft | **0.20** | 0.40 | стабильность оценённого лага (stdev < 30% от mean); warmup=0 |
| `affinity_threshold` | — | — | 0.5 |  |
| `affinity_stable_min` | — | — | 5 |  |

### 3.3. Предусловия (раннтайм-проверки на каждый план)

- Тикер допущен стратегией в `TickerUniverse` (`is_tradable_by(t, "leaderlag")`)
- Есть настроенный `LeaderTracker` для тикера
- Rolling correlation за 60 сек ≥ `leader_min_correlation` (**0.6**)
- Не торговать в первые 3 сек после открытия сессии / крупной новости
  (волатильность ломает корреляцию)

### 3.4. Условия входа

| Условие | Сигнал/метрика | Порог |
|---------|----------------|-------|
| C1 | `LeaderMove` signal | `leader_change_5s >= 0.3%`, `lag_pct >= 0.15%` |
| C2 | Возраст сигнала ≤ `lag_max_age_ms` | **3000 ms** |
| C3 | Rolling correlation 60-сек | `\|corr\| >= 0.6` |
| C4 | Своё движение ещё не началось (мы не опоздали) | `\|our_change_2s\| <= 0.1%` |
| C5 | На пути в сторону движения НЕТ крупной плотности в пределах 25 bps (иначе гэп не закроется) | нет `DensityDetected` на пути |
| C6 | Спред не раздут | `spread_bps <= 3` |

### 3.5. Параметры TradePlan

- **entry_type** = `Market` (скорость — суть стратегии)
- **stop_price** = ближайший локальный экстремум ИЛИ `entry_price ∓ stop_distance_bps` (default **8 bps** — это короткий стоп)
- **tp1_price** = `entry_price ± expected_catchup_bps * 0.7` где `expected_catchup_bps = |lag_pct_percent| * 100 * sign(corr) * direction` — то есть целим в 70% от ожидаемого догона.

  **Численный пример.** `leader_change_5s = +0.42%`, `our_change_5s = +0.08%`, `corr_60s = +0.78`. Тогда `expected_change_percent = 0.42 * 0.78 = 0.328%`, `lag_pct_percent = 0.328 - 0.08 = 0.248%`, направление — long (`lag > 0`), `expected_catchup_bps = 0.248 * 100 = 24.8 bps`, `tp1_offset_bps = 24.8 * 0.7 = 17.4 bps`. При `stop_distance_bps = 8` R:R = 17.4/8 ≈ 2.2.
- **tp2_price** — если поводырь продолжает идти в нашу сторону, тянем TP2 trailing-стопом на 0.3% за ценой
- **tp1_size_ratio** = 0.6 (забираем больше на первой цели, т.к. догон быстрый)

### 3.6. Инвалидация до входа

- Поводырь развернулся (`leader_change_1s` сменил знак)
- Наш инструмент уже прошёл ≥ 60% от `expected_catchup` — опоздали
- Возраст сигнала > 3 сек

### 3.7. Инвалидация после входа

- Корреляция сломалась (rolling corr упала ниже 0.5 за 10 сек) → закрыть по рынку
- Поводырь развернулся и прошёл > 0.3% против нас → закрыть по рынку

### 3.8. Типичные ошибки

- **Новостной шум**: если альт двигается по своим новостям, корреляция рушится, стратегия не работает. Поэтому C3 — rolling corr, а не статическая корреляция.
- **Крупная плотность на пути** (C5). Гэп не закроется — догона не будет.

## 4. FlushReversal — Прострел и возврат (опционально, Фаза 5)

### 4.1. Идея

Классический паттерн из мануала: один крупный принт выбрасывает цену на
N bps, маркетмейкеры возвращают её обратно. Ловим возврат.

**Правило из мануала:** единичный прострел не торгуем. Торгуем, если
наблюдали **уже 2 прострела в том же диапазоне** — ставим лимитки
"примерно до куда стреляло".

### 4.2. Профиль кандидата (affinity) — `[universe.affinity.flush]`

| Критерий | Kind | Weight | Default | Зачем |
|----------|------|--------|---------|-------|
| `min_volume_24h_usd` | **hard** | — | $50M | маркетмейкеры должны быть |
| `max_avg_spread_bps` | **hard** | — | 6 | прострелы часто на менее ликвидных альтах |
| `min_flush_events_per_day` | soft | **0.40** | 3 | паттерн «прострел» характерен |
| `flush_return_rate_7d` | soft | **0.40** | 0.60 | ≥60% прострелов возвращались в mid_before_flush за 10 сек; warmup=0 |
| `liquidation_availability` | soft | **0.20** | true | доступен liquidations feed (ARCH § 2.12) для LiquidationDetector |
| `affinity_threshold` | — | — | 0.4 |  |
| `affinity_stable_min` | — | — | 5 |  |

**Замечание по max_avg_spread_bps=6:** в § 5 глобальный `max_spread_bps=3`.
Здесь разрешён более широкий спред, но на **runtime-проверке в § 4.3** уже
работает глобальный фильтр `max_spread_bps` (или его strategy-override).
То есть флэш-тикер может попасть в pool с avg_spread 5 bps, но конкретный
TradePlan будет отбит, если в момент входа спред > 3 bps. Для смягчения
этого противоречия в § 5 добавлен strategy-level `max_spread_bps_override`.

### 4.3. Предусловия (раннтайм-проверки на каждый план)

- Тикер допущен стратегией в `TickerUniverse` (`is_tradable_by(t, "flush")`)
- Зафиксировано **≥ 2 прострела** (`TapeFlush`) в том же направлении в пределах 20 bps друг от друга за последние `flush_window_sec` (default **120 сек**)
- Ликвидность рынка средняя/высокая (иначе возврата может не быть)

### 4.4. Условия входа

- Цена снова прошла через прежнюю зону прострела (± 5 bps)
- Сразу после прохода — агрессия НЕ продолжается (`TapeFade`)
- Объём принта, ушедший в прострел, сопоставим с предыдущими (`flush_size >= 0.6 × avg_previous_flush_size`)
- **Прострел подтверждён `LiquidationFlush`** (см. SIGNAL_DETECTION § 9): есть ликвидационные ордера Binance в корреляционном окне ±3 сек от flush-принта. Это отличает капитуляцию (торгуем) от шумного одиночного принта (не торгуем).

### 4.5. Параметры TradePlan

- **entry_type** = `Limit`, цена чуть глубже предыдущих прострелов
- **stop_price** = за уровнем N-го прострела + buffer
- **tp1_price** = возврат к `mid_before_flushes` (обычно даёт 1.5–3R)
- **tp1_size_ratio** = 0.7 (берём большую часть быстро — возврат короткий)
- **tp2** обычно нет — стратегия короткая

### 4.6. Почему это Фаза 5, а не Фаза 3

Стратегия требует хорошего контекста (история прострелов), более сложного
ApproachAnalyzer'а, точной классификации принтов, **внешнего фида
ликвидаций Binance** (см. ARCHITECTURE § 2.12 — `external_feeds.liquidations`),
**LiquidationDetector** (SIGNAL § 9). Реализуется после того, как базовые
3 стратегии показали стабильность.

## 6. Матрица стратегия × условие рынка

| Рынок | Отскок (§ 1) | Пробой (§ 2) | LeaderLag (§ 3) | FlushReversal (§ 4) |
|-------|-------------|-------------|-----------------|---------------------|
| Тренд с BTC-поддержкой | ⚠️ (только против микро-откатов) | ✅ по тренду | ✅ | ✅ |
| Боковик / флэт | ✅ (идеально) | ⚠️ (часто ложные) | ⚠️ (нет импульса поводыря) | ✅ |
| Новостной всплеск | ❌ | ⚠️ (только в сторону новости) | ❌ (корреляция ломается) | ❌ |
| Низкая ликвидность | ❌ | ❌ | ❌ | ❌ |

✅ работаем, ⚠️ — только при идеальной композиции, ❌ — блокировано.

Реализация фильтра рынка — `StrategyEngine.classify_regime()` — через
**3-state HMM** (Trend / Range / News) с observations:
`[trend_slope_60s, volatility_1min_bps, leader_change_5s_pct]`. Forward
algorithm даёт posterior probability на каждый state, argmax выдаёт
текущий регим. Преимущество над if-else порогами: smooth state transitions
(нет «дрожания» между Trend/Range на granice), явная uncertainty
(`P(state)` ∈ [0,1]) — стратегия может требовать `posterior >= 0.7` для
своей активации. Emission-параметры (mean/cov per state) — labeled
training из исторических FeatureFrame snapshots (отдельный tool
`tool/regime_hmm_train/`, см. ARCHITECTURE § 12 algorithm catalog).
Fallback при отсутствии trained-model — наивная регрессия + порог по vol.

## 7. Приоритет стратегий при одновременных сигналах

Если на одном тикере одновременно сгенерированы несколько TradePlan —
приоритет фиксированный (см. `strategy_priority` в конфиге):

1. **LeaderLag** — самая узкая временная возможность, берём первой
2. **BreakoutEatThrough** — второй
3. **BounceFromDensity** — третий
4. **FlushReversal** — последний

Если TradePlan с более высоким приоритетом прошёл RiskManager — план с
низшим приоритетом отбрасывается на этом тикере.

## 8. Журнал (обязательно для каждой сделки)

Для каждой сделки `TradeJournal` сохраняет:
- все входные Signal'ы (evidence)
- снимок FeatureFrame на момент входа
- пороги, которые использовались (из конфига)
- результат: PnL, holding time, cause_of_exit (tp1/tp2/stop/timeout/manual)

Это база для пост-мортема. Скальпинг невозможен без разбора своих сделок —
одно из главных правил мануала.
